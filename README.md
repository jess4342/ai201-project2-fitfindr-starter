# FitFindr

FitFindr is a secondhand shopping assistant that searches mock marketplace listings, recommends outfits from a user wardrobe, and generates a shareable fit card. The project connects three tools through a planning loop and exposes them through a Gradio web interface.

## Tool Inventory

### 1. `search_listings(description: str, size: str | None, max_price: float | None) -> list[dict]`
- Inputs:
  - `description`: a cleaned natural-language query describing the desired item
  - `size`: optional size filter (e.g. `"M"`, `"L"`, `"XS"`)
  - `max_price`: optional upper bound on price
- Output:
  - a list of matching listing dictionaries from `data/listings.json`
- Purpose:
  - Find the best marketplace item that matches the user request, size, and budget.

### 2. `suggest_outfit(selected_item: dict, wardrobe: dict, trend_context: str | None, style_profile: dict | None) -> str`
- Inputs:
  - `selected_item`: the chosen listing dictionary from search results
  - `wardrobe`: the user's wardrobe dict from `utils/data_loader.py`
  - `trend_context`: optional trend information that should influence styling
  - `style_profile`: optional remembered style preferences from prior interactions
- Output:
  - a natural-language outfit suggestion string
- Purpose:
  - Use the selected item, wardrobe, trend context, and remembered preferences to recommend a wearable outfit.

### 3. `compare_price(selected_item: dict, listings: list[dict]) -> str`
- Inputs:
  - `selected_item`: the chosen listing dictionary
  - `listings`: the full listings dataset
- Output:
  - a price assessment string comparing the selected item to similar listings
- Purpose:
  - Assess whether the item is a good deal or a splurge based on comparable items.

### 4. `get_trend_insight(selected_item: dict) -> str`
- Inputs:
  - `selected_item`: the chosen listing dictionary
- Output:
  - a short trend insight string based on category and style tags
- Purpose:
  - Provide a trend-aware styling signal that influences the outfit suggestion.

### 5. `create_fit_card(outfit_suggestion: str, selected_item: dict) -> str`
- Inputs:
  - `outfit_suggestion`: text generated by `suggest_outfit`
  - `selected_item`: the selected listing dictionary
- Output:
  - a short, social-friendly fit card string summarizing the outfit and the purchase
- Purpose:
  - Summarize the final result with shareable conversational copy.

## Planning Loop

The agent is built as a deterministic planning loop inside `agent.py`. It is not a single monolithic prompt; it is a sequence of decision steps that pass state explicitly.

The loop works like this:

1. Initialize a fresh session dictionary with `_new_session(query, wardrobe)`.
2. Parse the raw user query to extract:
   - a cleaned `description`
   - optional `size`
   - optional `max_price`
3. Call `search_listings()` with those parsed values.
4. If search returns no listings, attempt a retry with loosened constraints (remove size or relax price) and store a `fallback_note` if that succeeds.
5. Otherwise select the top result and store it in `session["selected_item"]`.
6. Enrich the selected item with `trend_insight` and `price_assessment`.
7. Call `suggest_outfit(selected_item, wardrobe, trend_context, style_profile)` and store the result in `session["outfit_suggestion"]`.
8. Call `create_fit_card(outfit_suggestion, selected_item)` and store the result in `session["fit_card"]`.

That ordering ensures the agent only generates outfit text once there is a valid listing, and only generates a fit card once there is a valid outfit suggestion.

## State Management

The agent uses a single session dict as the shared state bag. This keeps the handoff between tools explicit and inspectable.

Key session fields:

- `query`: original user text
- `parsed`: extracted `description`, `size`, and `max_price`
- `search_results`: list of candidate listings
- `selected_item`: top listing chosen for styling
- `wardrobe`: the user wardrobe passed in from the UI
- `trend_insight`: trend information added for the selected item
- `price_assessment`: price comparison reasoning returned by the stretch tool
- `fallback_note`: explanation of any retry / loosened search behavior
- `style_profile`: remembered style preferences carried across interactions
- `outfit_suggestion`: string returned by `suggest_outfit`
- `fit_card`: final string returned by `create_fit_card`
- `error`: early-return message if any step fails

Using a session dictionary means every intermediate result is available for debugging and for later extension.

## Error Handling

Error handling is explicit at each step so the agent fails gracefully.

- `search_listings` failure:
  - If no marketplace item matches the parsed query, size, or budget, `run_agent()` first tries a fallback search with loosened constraints.
  - If fallback still fails, it sets `session["error"]` and returns immediately.
  - Example: query `designer ballgown size XXS under $5` produces:
    - `No listings matched your description, size, and price. Try broadening your query or removing the size or price filter.`

- `suggest_outfit` failure:
  - If outfit generation returns an empty or invalid string, the agent stops and reports:
    - `I couldn't create outfit suggestions for that item right now. Please try again later.`

- `create_fit_card` failure:
  - If fit-card generation fails or returns blank, the agent reports:
    - `I couldn't create a fit card from that outfit suggestion. Please try again or search for a different item.`

The UI maps `session["error"]` to the first panel and clears the other two panels when an error occurs.

When a fallback search succeeds after a zero-result query, the listing output also includes a note explaining which constraint was loosened.

## Stretch Features

### Price Comparison Tool

The agent now uses `compare_price()` to assess whether the selected listing is a good deal. It compares the item's price to similar listings in `data/listings.json` by category and shared style tags. The output is shown in the listing panel as a `Price assessment:` note.

Example behavior:
- If the item is priced well below comparables, the note says it looks like a good deal.
- If the item is near the average price, it says it feels reasonable.
- If the item is above average, it warns that the item may be a splurge.

### Trend Awareness Tool

The agent uses `get_trend_insight()` to generate a trend-aware styling signal based on the item's category and style tags. This tool uses the mock listing style tags and a fixed trend mapping to decide what kind of modern styling advice to give.

Data source:
- `data/listings.json` style tags and item category
- a static internal mapping of tag → trend insight

The trend note is passed into `suggest_outfit()` so the generated outfit can reflect what is currently fashionable.

### Style Profile Memory

The agent maintains a lightweight style preference profile across interactions using a global memory object in `agent.py`.

How it works:
- After each successful run, `_build_style_profile()` extracts themes from the query and the selected item's style tags.
- Those preferences are stored in `STYLE_PROFILE`.
- On later runs, the agent passes `style_profile` into `suggest_outfit()` so the wardrobe recommendations can stay aligned with earlier preferences.

This means the same app session can remember a user's earlier taste without asking them to re-enter it.

### Retry Logic with Fallback

If an initial search returns zero results, the agent automatically retries with loosened constraints:
- first by removing the size filter if there was one,
- then by increasing the budget,
- then by searching across all sizes and prices.

If retry succeeds, the UI displays a note explaining the adjustment.

## App Flow

`app.py` connects the Gradio interface with the agent loop via `handle_query()`:

1. Validate that the user entered text.
2. Choose `get_example_wardrobe()` or `get_empty_wardrobe()` based on the radio selection.
3. Call `run_agent(user_query, wardrobe)`.
4. If `session["error"]` is set, return it in the first output panel and empty strings for the other two.
5. Otherwise format `session["selected_item"]` into readable `listing_text`, then return:
   - `listing_text`
   - `session["outfit_suggestion"]`
   - `session["fit_card"]`

This ensures the UI always shows either a successful three-panel result or a clean error state.

## Spec Reflection

This implementation follows the spec by making the agent decision process explicit rather than implicit. The planning loop is not just a series of tools; it is a dataflow design where:

- parsed query state feeds search,
- search results feed outfit suggestion,
- outfit suggestion feeds fit-card creation,
- errors short-circuit later steps.

That structure makes it clear what decisions the agent makes at each step, and it guarantees that the UI only displays complete results when all three tools have successfully produced output.
